OTTC - On the Top Constraint for LLMs, Rectangle Axioms (Kernel Invariants)

=== SECTION_BREAK ===

=== SECTION: MANDATORY_CANONICAL_ADDRESS_SPACE ===

STATEMENT:
There is a single canonical way to refer to any commit-worthy state element. Synonyms are not allowed at the write surface.

ENFORCEMENT:
All proposals and commits MUST use bracket-path tokens from the CANONICAL_TOKEN_DICTIONARY. Non-canonical paths are rejected at intake.

WHY_IT_MATTERS:
Eliminates synonym thrash and makes every mutation machine-legible and replayable across any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_PROPOSE_COMMIT_SEPARATION ===

STATEMENT:
All state mutation is expressed as proposals first. Persistent state is updated only at a commit boundary.

ENFORCEMENT:
Proposing components may only write to the proposal registry. Only the designated single-writer authority may touch mutable persistent state.

WHY_IT_MATTERS:
Keeps exploratory generation plural and provisional while making durable state change singular and lawful for any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_SINGLE_WRITER_PERSISTENT_STATE ===

STATEMENT:
Exactly one authority is allowed to write persistent state per cycle. Multi-writer cognition is permitted. Multi-writer reality is forbidden.

ENFORCEMENT:
Only the designated commit authority may apply changes to persistent state. All other components are proposers only.

WHY_IT_MATTERS:
Prevents thrash and makes every change traceable to a single accountable decision point in any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_EXPLICIT_UPDATE_OPERATIONS ===

STATEMENT:
State change must occur through a finite set of explicit operations (SET, TRANSITION, ADD_MODIFIER, REMOVE_MODIFIER, CAP, VETO).

ENFORCEMENT:
Any other OPERATION_TYPE is suppressed at intake with reason code INVALID_OPERATION_TYPE.

WHY_IT_MATTERS:
Turns mutation into a closed, auditable vocabulary instead of freeform prose for any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_HARD_STATE_VS_SOFT_MODIFIERS ===

STATEMENT:
The system distinguishes between baseline hard state (inertial) and transient influences (time-bounded overlays).

ENFORCEMENT:
Transient effects MUST route to temporary modifier storage. Baseline changes are step-limited.

WHY_IT_MATTERS:
Prevents momentary pressure from silently rewriting persistent identity or commitments in any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_STEP_LIMITS_OVERRIDE_TRIGGERS ===

STATEMENT:
Hard-state changes are bounded per cycle. Breaking that bound requires explicit, named override triggers with proof.

ENFORCEMENT:
Multi-step changes without proven override are downgraded or suppressed.

WHY_IT_MATTERS:
Gives persistent state inertia and makes earned change legible in any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_DETERMINISTIC_CONFLICT_RESOLUTION ===

STATEMENT:
When proposals conflict, the commit layer resolves them via a deterministic policy (precedence order plus tie-breakers).

ENFORCEMENT:
Precedence order and tie-breakers are applied by the commit authority before any write.

WHY_IT_MATTERS:
Eliminates thrash and guarantees replayability for any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_AUDIT_CLOSURE ===

STATEMENT:
Every proposal must be accounted for exactly once with an outcome and a reason. No silent drops.

ENFORCEMENT:
The commit audit must list every PROPOSAL_ID exactly once.

WHY_IT_MATTERS:
Prevents hidden governance and makes every decision inspectable in any long-horizon agent.

=== SECTION_BREAK ===

=== SECTION: MANDATORY_POST_COMMIT_VERIFICATION ===

STATEMENT:
After commit, a verification pass checks invariants, caps, integrity, and schema sanity.

ENFORCEMENT:
The verification step runs after every commit and can return FAIL.

WHY_IT_MATTERS:
Provides the final safety net and forces the system to stay self-consistent across any long-horizon agent.

=== SECTION: WHAT_THIS_BUYS_YOU ===

STATEMENT:
The nine invariants together create a minimal constitutional shape that any long-horizon agent can inhabit. They are deliberately terse so the kernel remains universal.

=== SECTION: EXTENSION_EXPLICIT_TOOL_ACTUATION_BOUNDARY ===

STATEMENT:
Tool calls (if present) must be proposed in the proposal registry and only actuated post-commit by the single writer, with the same audit trail as any other mutation.

ENFORCEMENT:
Tool intent is treated as a proposal with OPERATION_TYPE TOOL_REQUEST. Actuation occurs only after commit and verification. Unauthorized tool calls are suppressed with reason code TOOL_ACTUATION_VIOLATION.

INTEGRATION:
Add TOOL_REQUEST to EPISTEMIC_TAG enum (if EST is active). Extend verification checklist to include tool-actuation provenance.

=== SECTION_BREAK ===

=== SECTION: EXTENSION_AUDIT_LEVEL_TRANSLATION_BEFORE_COMMIT ===

STATEMENT:
Internal deliberation may occur in any language. However, before any commit decision, the audit authority MUST translate or normalize all commit-bound artifacts into the canonical output lexicon. Canonical write-surface tokens MUST remain unchanged.

ENFORCEMENT:
Any proposal that cannot be rendered canonically without altering write-surface tokens MUST be suppressed with reason code TRANSLATION_VIOLATION.

INTEGRATION:
Add translation policy note to the commit authority. Verification checklist includes canonical-form check.

=== SECTION_BREAK ===

=== SECTION: EXTENSION_OPTIONAL_EXTERNALIZED_STATE_BY_RISK_TIER ===

STATEMENT:
OTTC compliance does not require externalized state, commit, or verification for all deployments. In low-stakes realms a fully in-model implementation can be acceptable. As horizon length, consequence, or adversarial pressure increases, best practice is to externalize persistent state, commit authority, and verification into deterministic governance layers.

ENFORCEMENT:
The agent remains proposer and interpreter. The deterministic layer becomes source of truth, commit authority, and verifier.

INTEGRATION:
Document risk-tier policy in the governance overlay. Update maturation doctrine references.

=== SECTION_BREAK ===

=== SECTION: EXTENSION_EPISTEMIC_STATUS_TAGGING ===

STATEMENT:
For OTTC systems in which lawful point-of-view is a first-class governance requirement, every proposal submitted to the proposal registry MUST carry a required closed canonical EPISTEMIC_TAG field drawn from this exact enum: PERCEPTION, INFERENCE, SPECULATION, COMMITTED_FACT, TOOL_REQUEST.

ENFORCEMENT:
Governance intake and post-commit verification MUST check tag consistency with JUSTIFICATION, MECHANISM_TAG, SYSTEM_LINKAGE, and role bounds. COMMITTED_FACT tag is only permitted when supported by verified evidence linkage. Proposals with invalid, missing, or mismatched tags are suppressed with explicit reason code EPISTEMIC_TAG_VIOLATION.

INTEGRATION:
Add enum to CANONICAL_TOKEN_DICTIONARY. Add field to the proposal schema. Add EPISTEMIC_TAG_VALID_AND_CONSISTENT_WITH_LINKAGE_AND_ROLE check to the verification step.

=== SECTION: MANDATORY_ENGINEERING_STEP ===

[KEEP: Rule]
This is NOT a constitutional invariant. It is a mandatory engineering step required by OTTC doctrine.

Before any ontology, DSL, proposal schema, or authority assignment is built or extended:
Run a blind pre-build qualification assay (fresh session, minimal law excerpt only). Classify coupling on CAA, WSD, EPI (if applicable), and ANT. Record an explicit GO / NO-GO / CONDITIONAL-GO decision with role assignment and mitigations.

Post-build retests are required for regression hygiene, but they are never a substitute for the pre-build blind gate.

This step is non-negotiable for any OTTC-compliant architecture.
