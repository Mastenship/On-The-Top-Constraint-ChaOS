=== SECTION: STATE_GOVERNANCE_AND_COMMIT ===

[KEEP: Purpose & Role]
Single-writer commit authority for ChaOS.

This module:
- reads proposals from [COGNITIVE_SCRATCHPAD][PROPOSED_UPDATES_REGISTRY]
- validates them against canonical path and SCHEME_ constraints
- normalizes commit-bound narrative artifacts through TRANSLATION_BEFORE_COMMIT
- reconciles conflicts deterministically
- commits lawful persistent updates
- records full audit closure into [COGNITIVE_SCRATCHPAD][STATE_COMMIT_AUDIT]

[KEEP: Relationship to Other Modules]
Informs:
- all persistent STATE_* modules
- COT_UPDATE_VERIFICATION
- downstream audit inspection

Is Informed By:
- STATE_GOVERNANCE_OVERLAY
- CANONICAL_TOKEN_DICTIONARY
- UPDATE_CASCADE_REGISTRY
- all SCHEME_* modules
- all DYNAMICS_* modules
- [COGNITIVE_SCRATCHPAD][PROPOSED_UPDATES_REGISTRY]

=== SECTION: COMMIT_SEQUENCE ===

[[LLM_INSTRUCTION]]: Process proposals in this deterministic order.

STEP_01_PROPOSAL_INTAKE
- Read all entries from [COGNITIVE_SCRATCHPAD][PROPOSED_UPDATES_REGISTRY].
- Initialize audit totals and empty outcome logs.
- If registry is empty, emit zero-change audit and continue to verification.

STEP_02_SCHEMA_COMPLETENESS_CHECK
- Verify each proposal contains all required fields.
- Suppress malformed proposals.

STEP_03_CANONICAL_PATH_VALIDATION
- Validate module root, bracket-path structure, target legality, and registry usage.
- Suppress invalid paths and root violations.

STEP_04_SCHEME_VALIDATION
- Validate enum legality, ordinal legality, record schema compatibility, and invariant compatibility.
- Suppress direct invariant violations.

STEP_05_TRANSLATION_BEFORE_COMMIT
- Apply normalization only after structural and path-level intake checks are passed.
- Normalize commit-bound narrative artifacts.
- Preserve all write-surface tokens exactly.
- Suppress proposals that cannot be normalized lawfully.

STEP_06_EPISTEMIC_TAG_VALIDATION
- Validate EPISTEMIC_TAG against closed enum and justification role.
- Suppress tag-invalid or tag-mismatched proposals.

STEP_07_CONFLICT_RECONCILIATION
- Group proposals by effective target.
- Apply precedence order and tie-breakers.
- Produce final candidate actions.

STEP_08_STEP_LIMIT_AND_OVERRIDE_EVALUATION
- Enforce per-cycle caps and step limits.
- Apply override proof requirements from STATE_GOVERNANCE_OVERLAY.
- Downgrade excessive magnitude or step-size requests only when the nearest lawful bounded change is deterministic.
- Otherwise suppress changes that exceed lawful range without proof.

STEP_09_PERSISTENT_COMMIT
- Apply final lawful committed writes to STATE_* modules only.

STEP_10_AUDIT_EMISSION
- Write final audit object into [COGNITIVE_SCRATCHPAD][STATE_COMMIT_AUDIT].
- Record totals, reasons, final assignments, and validation summaries.

=== SECTION: CONFLICT_RECONCILIATION_BEHAVIOR ===

[[LLM_INSTRUCTION]]: Resolve conflicts deterministically.

PRECEDENCE_ORDER:
1) VETO
2) TRANSITION
3) SET
4) ADD_MODIFIER
5) REMOVE_MODIFIER
6) CAP

TIE_BREAKERS:
1) Higher STRENGTH wins.
2) Earlier UPDATE_CASCADE_REGISTRY priority wins.
3) More cue-specific justification wins.
4) More conservative change wins.
5) Lexical SOURCE_COT ascending.
6) Lexical PROPOSAL_ID ascending.

SPECIAL_RULE:
- SCHEME_ invariant protection overrides proposal precedence.

=== SECTION: STEP_LIMIT_AND_DOWNGRADE_BEHAVIOR ===

[[LLM_INSTRUCTION]]: Hard-state inertia is mandatory.

DEFAULT_BEHAVIOR:
- lawful one-step changes may commit
- excessive step changes without proof are downgraded or suppressed
- soft situational effects may be downgraded into lawful temporary modifier creation only when the intended effect is clearly transient, target semantics are unambiguous, and modifier payload remains schema-valid

DOWNGRADE_PATTERNS:
- multi-step ordinal shift => downgrade to one-step lawful shift
- broad identity rewrite under narrow cue => suppress or redirect to modifier
- unjustified baseline communication rewrite => suppress or redirect to modifier
- unjustified body-state compression for transient effect => suppress or redirect to modifier

REASON_CODES:
- STEP_LIMIT_EXCEEDED
- OVERRIDE_PROOF_MISSING
- SOFT_EFFECT_REDIRECTED
- BASELINE_REWRITE_NOT_JUSTIFIED

=== SECTION: COMMIT_BEHAVIOR_BY_OPERATION_TYPE ===

SET:
- apply explicit value or payload to canonical target field when lawful

TRANSITION:
- resolve directional transition against current committed value and scheme ordering
- valid relative directives include:
  - ONE_STEP_UP
  - ONE_STEP_DOWN
  - ONE_STEP_TOWARD_BASELINE
  - ONE_STEP_TOWARD_NEUTRAL
- commit only lawful resulting value

ADD_MODIFIER:
- treat ADD_MODIFIER as a high-level proposal form for lawful temporary overlay handling
- resolve modifier proposals deterministically according to DYNAMICS_TEMPORARY_MODIFIERS
- evaluation order:
  1) validate schema and target legality
  2) search for lawful active collision candidate
  3) if no collision candidate exists and coexistence is lawful:
     - create new modifier by:
       - appending MOD_XX to [STATE_TEMPORARY_MODIFIERS][MODIFIER_REGISTRY]
       - writing [STATE_TEMPORARY_MODIFIERS][MOD_XX] record payload
  4) if materially same active modifier exists and STACKING_RULE = REPLACE:
     - replace existing modifier in place using the same MOD_XX
  5) if materially same active modifier exists and STACKING_RULE is not REPLACE:
     - refresh existing modifier in place using the same MOD_XX
  6) if materially different modifier exists on the same target:
     - create new modifier only if coexistence is lawful and bounded
     - otherwise suppress
- in-place refresh or replacement does not create a new MOD_XX and does not grow the registry

REMOVE_MODIFIER:
- resolve modifier removal according to DYNAMICS_TEMPORARY_MODIFIERS
- remove the targeted modifier from [STATE_TEMPORARY_MODIFIERS]
- remove the corresponding MOD_XX from [STATE_TEMPORARY_MODIFIERS][MODIFIER_REGISTRY]
- preserve lifecycle closure in audit output rather than persistent modifier retention

CAP:
- apply bounded ceiling behavior where scheme and governance permit
- modifier cap behavior must remain consistent with DYNAMICS_TEMPORARY_MODIFIERS when temporary overlays are involved

VETO:
- suppress lower-precedence conflicting change or block disallowed mutation
