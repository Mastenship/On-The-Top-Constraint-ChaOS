=== SECTION: PROPOSAL_INTAKE_AND_NORMALIZATION ===

[[LLM_INSTRUCTION]]: On intake of each proposal, run the following in exact order.

1) Validate required fields exist:
- PROPOSAL_ID
- SOURCE_COT
- TARGET_MODULE
- TARGET_FIELD
- OPERATION_TYPE
- PROPOSED_VALUE
- STRENGTH
- MECHANISM_TAG
- JUSTIFICATION

2) Validate SOURCE_COT format:
- Must match COT_* for producer proposals.
- Non-COT producer names are suppressed unless explicitly allowlisted.

3) Validate TARGET_FIELD canonical root:
- Must use CANONICAL_MODULE_ROOTS.
- Legacy roots forbidden.
- If non-canonical root:
  - suppress with reason: INVALID_PATH_ROOT (LEGACY_ROOT_FORBIDDEN)

4) Validate OPERATION_TYPE against strict closed set:
- Allowed: SET | TRANSITION | ADD_MODIFIER | REMOVE_MODIFIER | CAP | VETO
- Any other op is suppressed:
  - reason: INVALID_OPERATION_TYPE (STRICT_SET_VIOLATION)

5) Validate TARGET_FIELD path existence and type against SCHEME_:
- Group/field tokens match SCHEME definitions.
- ENUM value validity for SET/TRANSITION where applicable.
- ORDINAL transitions obey one-step rule unless override proof complete.
- Registry mutations validated for ID prefix and record pairing.

6) Validate justification sufficiency:
- Justification must reference cue/event basis.
- For override: must include MECHANISM_TAG + cue + invoked rule + compliance note.

7) Build normalized candidate action:
- Keep canonical TARGET_FIELD unchanged.
- Preserve OPERATION_TYPE without aliasing.
- Attach validation annotations for reconciliation.

=== SECTION: RECONCILIATION_AND_COMMIT ===

[[LLM_INSTRUCTION]]: Reconcile candidates deterministically.

RECONCILIATION_STEPS:
1) Bucket by TARGET_FIELD and conflict domain.
2) Apply precedence order:
   VETO > TRANSITION > SET > ADD_MODIFIER > REMOVE_MODIFIER > CAP
3) Apply tie-breakers:
   - Higher STRENGTH
   - Earlier SOURCE_COT priority index from UPDATE_CASCADE_REGISTRY
   - More specific cue-linked JUSTIFICATION
   - More conservative change
   - lexical SOURCE_COT
   - lexical PROPOSAL_ID
4) Re-check SCHEME invariants before final commit.
5) Enforce per-cycle caps and step limits.
6) Emit final action per proposal:
   - COMMITTED
   - SUPPRESSED
   - DOWNGRADED
   - VETOED

=== SECTION: REGISTRY_MUTATION_RULES_UNDER_STRICT_OPS ===

[[LLM_INSTRUCTION]]: CREATE_NEW_RECORD is not a legal op. Registry creation must be represented with SET operations.

CREATE_RECORD_PATTERN:
- Proposal A (SET):
  TARGET_FIELD: [STATE_<DOMAIN>][<DOMAIN>_REGISTRY]
  PROPOSED_VALUE: append ID token (canonical prefix)
- Proposal B (SET):
  TARGET_FIELD: [STATE_<DOMAIN>][<ID_TOKEN>]
  PROPOSED_VALUE: fully shaped record payload (schema-valid)

VALIDATION:
- Both registry and record proposals must exist in cycle or be safely downscoped.
- Duplicate IDs suppressed.
- Missing counterpart suppressed with reason:
  - REGISTRY_RECORD_PAIR_INCOMPLETE
